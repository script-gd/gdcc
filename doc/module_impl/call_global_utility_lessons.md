# CALL_GLOBAL Utility 实现复盘与经验教训

## 1. 背景

本复盘针对上一版 `CALL_GLOBAL` utility 调用实现。该实现已删除，本文用于沉淀可复用的工程经验，避免在后续重建时重复踩坑。

## 2. 关键问题归纳

### 2.1 过度耦合到 `CBodyBuilder`

- 在 Builder 中一次性塞入过多 utility 专有逻辑，导致通用调用语义和特化语义边界模糊。
- 后续修改 utility 行为时，容易影响非 utility 调用路径。

### 2.2 复用不足，出现重复实现

- 旧实现没有优先复用既有赋值/生命周期 API，而是在局部路径重复实现同类语义。
- 结果是“同一语义多处实现”，修一个点会漏另一个点。

### 2.3 手工字符串拼接过多

- 大量拼接 C 调用片段（含参数、返回赋值、临时变量）使可读性和可验证性下降。
- 一旦涉及对象生命周期或指针转换，手拼字符串很容易引入隐性 bug。

### 2.4 校验职责分层不清晰

- 指令生成器与 Builder 的校验职责存在重叠和空洞。
- 错误信息在不同路径上风格不一致，不利于定位问题。

### 2.5 文档与实现漂移

- 文档先行固化了大量“目标态”细节，但代码仍在重构中。
- 随着实现回滚，文档没有同步降级，造成“文档看起来已完成、实际并未稳定”的误导。

## 3. 这次删除带来的明确结论

- utility 调用相关设计不应继续以“补丁式追加”推进。
- 在没有稳定 API 之前，不要把临时实现写成长期规范文档。
- 代码结构优先于覆盖面：先建立清晰边界，再扩展功能。

## 4. 后续重建的约束建议

### 4.1 先定义边界，再写代码

- `CallGlobalInsnGen` 负责：指令级解析、变量存在性检查、结果变量合法性检查。
- `CBodyBuilder` 负责：调用发射与通用生命周期语义。
- `CGenHelper` 负责：名称规范化、类型渲染、通用片段生成。

### 4.2 先复用现有 API，禁止复制语义

- 已有赋值/返回值处理路径必须复用，不新增“平行实现”。
- 若现有 API 不足，先抽象补齐 API，再接入 `CALL_GLOBAL`。

### 4.3 严控字符串拼接

- 禁止在指令生成器里手拼复杂调用表达式和生命周期逻辑。
- 优先通过 `ValueRef` / `TargetRef` / helper 方法表达语义。

### 4.4 文档策略改为“双层”

- `doc/gdcc_c_backend.md` 只记录稳定契约。
- `doc/module_impl/*.md` 记录阶段性方案，并显式标注状态（草案/实施中/归档）。

## 5. 最小可行重建路线（建议）

1. 先完成 `CALL_GLOBAL` 的最小 non-vararg utility 路径，且仅覆盖变量参数。
2. 复用现有返回值赋值与类型兼容检查，不新增特化赋值流程。
3. 增加 focused tests（成功路径 + 关键失败路径），再扩展 vararg/default。
4. 每扩展一块语义，先更新 module_impl 状态文档，再更新稳定文档。

## 6. 评审检查清单

- 是否复用了已有 API，而非复制逻辑？
- 是否引入了新的手工字符串拼接热点？
- 是否明确了生成器/Builder/helper 的职责边界？
- 错误消息是否可定位（函数/块/指令）且风格统一？
- 文档状态是否与当前代码一致？
