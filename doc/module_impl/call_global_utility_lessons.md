## 1. 文档用途

本文仅保留 CALL_GLOBAL utility 路径的长期设计经验，用于代码评审和后续重构时避坑。  
任何“是否已完成”的问题以代码与测试、以及实施状态文档为准。

## 2. 历史问题抽象（可复用）

### 2.1 职责混叠

- 把 utility 专有语义大量塞进 `CBodyBuilder`，会模糊通用调用语义边界。
- 结果是修改 utility 行为时，容易波及非 utility 路径。

### 2.2 语义重复实现

- 未复用已有赋值/生命周期 API，导致同类语义多处实现。
- 修复时容易“修一处漏一处”。

### 2.3 过度手工拼接

- 在生成器手拼复杂调用表达式（参数、返回、临时变量）可读性差且难验证。
- 涉及对象生命周期和指针转换时，风险显著上升。

### 2.4 校验分层不清晰

- 生成器与 Builder 的校验职责重叠或缺失，会导致错误信息风格和触发时机不一致。

## 3. 固化约束（长期有效）

### 3.1 先划边界，再扩能力

- `CallGlobalInsnGen`：负责指令级解析与语义校验。
- `CBodyBuilder`：负责通用发射、参数渲染与生命周期语义。
- `CGenHelper`：负责名称规范化与共享辅助逻辑。

### 3.2 先复用，再扩展

- 优先复用 `callVoid` / `callAssign` 的既有语义。
- 若能力不足，先补齐通用 API，再接入新语义；禁止平行实现。

### 3.3 vararg 契约统一由 Builder 承担

- `varargs == null`：跳过 vararg 尾参数。
- `varargs != null`：始终生成 vararg 尾参数（空列表发射 `NULL, (godot_int)0`）。
- `argv` 组装逻辑归属 `CBodyBuilder.renderVarargArgv(...)`，避免生成器手工拼接。

### 3.4 默认参数策略需单点实现

- 当前默认参数补全尚未实现时，可在生成器侧明确抛出 `NotImplementedException`。
- 后续实现时，应将 default 物化逻辑集中到单点，避免多处解析漂移。

## 4. 评审检查清单

- 是否复用了已有 API，而不是复制语义？
- 是否把 utility 专有逻辑错误地下沉到 Builder？
- 是否新增了手工字符串拼接热点？
- 错误消息是否保持可定位、风格一致？
- 进度状态是否已同步到实施状态文档？
